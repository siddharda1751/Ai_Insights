This project was designed as an asynchronous workflow automation system that simulates how AI-powered lead enrichment platforms operate in real production environments.

The core goal was to automate the complete lead intelligence pipeline:
- capturing lead information
- enriching company data through web scraping
- generating AI-powered business insights
- creating personalized PDF audit reports
- delivering reports automatically through email

One of the key architectural decisions was separating the application into independent services:
- scraping service
- AI insight generation service
- PDF generation service
- email delivery service
- Google Sheets tracking service

This service-oriented structure improved maintainability, readability, and workflow orchestration.

Another important design decision was using asynchronous background execution instead of making the frontend wait for long-running operations such as scraping, AI generation, and PDF rendering. The API immediately returns a success response while the workflow continues independently in the background. This approach improves responsiveness and reflects how production automation systems are commonly designed.

Initially, the system used SMTP-based Gmail delivery through Nodemailer. The implementation worked correctly in local development. However, during cloud deployment testing on Render, outbound SMTP communication consistently failed due to infrastructure/network restrictions associated with SMTP socket connections on the deployment environment.

Alternative SMTP configurations and API-based email approaches were also tested, but stable production deployment could not be completed within the assignment timeframe. The complete email delivery workflow was therefore demonstrated successfully in local execution.

Another major learning came from attempting Google Drive persistence for generated reports. The initial plan was to upload generated PDFs to Google Drive and store public links inside Google Sheets. However, Google Service Accounts do not provide normal personal Drive storage quota. Properly supporting this would require OAuth delegation or Google Workspace Shared Drives, which introduced additional infrastructure complexity beyond the scope of the prototype.

As a result, I intentionally simplified the architecture into a stateless workflow using in-memory PDF buffers instead of persistent file storage. This reduced:
- authentication complexity
- storage management overhead

I also made the decision to use strict failure propagation throughout the workflow. If scraping, AI generation, or PDF generation fails, the workflow aborts entirely instead of generating fallback or low-quality reports. This was important because the assignment emphasized highly personalized and professional outputs rather than best-effort automation.

Retry mechanisms were implemented for:
- scraping
- AI API requests
- email delivery

to improve resilience against temporary failures and unstable external APIs.

Google Sheets was integrated as a lightweight operational dashboard to provide workflow observability and track real-time statuses such as:
- Scraping
- Generating AI Insights
- Generating PDF
- Sending Email
- Completed
- Failed

This project helped me better understand:
- asynchronous workflow orchestration
- service-oriented backend architecture
- AI-powered automation pipelines
- failure propagation strategies
- cloud deployment limitations
- third-party API integration
- operational observability design

If given more time, I would further improve the system by:
- introducing queue-based workers using BullMQ + Redis
- improving multi-page website intelligence extraction
- enhancing PDF visual design and branding
- implementing workflow analytics dashboards
- adding CRM integrations and scheduled follow-up automations
- supporting persistent cloud storage through OAuth-based Drive integration
